## 1. Project overview

A self-hosted, multi-user wardrobe management and outfit recommendation app. Users photograph their clothing, the app catalogs each item via AI vision, and a chatbot recommends outfits based on real-world scenarios (weather, activity, formality). The system learns from user choices over time.

**Design philosophy:** No database, no cloud dependency, no build tools. Everything runs on a single Mac, serves a web UI over the local network, and stores data as flat JSON files.

---

## 2. Current architecture

### Dual API design

The project intentionally separates two AI workloads across two providers:

|Workload|Provider|SDK|Model|When it runs|
|---|---|---|---|---|
|Cataloging (vision)|Anthropic|`anthropic` Python SDK|claude-sonnet-4-20250514|One-time per item (image → JSON label)|
|Recommendations (chat)|Alibaba DashScope (Qwen)|`openai` Python SDK (compatible endpoint)|qwen3-coder-plus|Every outfit request|

**Why two providers:** Vision cataloging requires strong image understanding (Claude excels here). Recommendations are text-only and run frequently, so a lower-cost provider (Qwen via existing company subscription) handles ongoing usage.

**Provider flexibility:** The recommendation endpoint uses the OpenAI-compatible SDK, so swapping to OpenAI, Groq, Together, or local Ollama requires only changing the base URL and model name in `config.py`. Cataloging is Anthropic SDK-specific and would need code changes to swap.

### Data flow

```
[iPhone camera roll]
    → Upload via browser (POST /api/upload/{username})
    → Saved to users/{username}/wardrobe_photos/

[Catalog button in UI or CLI]
    → Each new image: resize to max 1024px → base64 → Anthropic Vision API
    → Response: JSON with category, color, material, thickness, season, formality
    → Appended to users/{username}/wardrobe.json

[User types scenario in chat]
    → Backend extracts season + formality from message (Python heuristics, no API call)
    → Filters wardrobe.json to matching items
    → If >60 items remain, cap at 60 (balanced across categories)
    → Enrich each item with preference_score from preferences.json
    → Send compact JSON + message to Qwen API
    → Qwen returns structured outfit recommendations (1–3 outfits)
    → Frontend renders outfit cards with clothing images

[User picks "Choose This Outfit"]
    → POST /api/decide/{username}
    → Updates item_scores (per-feature affinity) and pair_scores (combo affinity)
    → Saved to preferences.json
    → Future recommendations biased toward preferred items + combos
```

### Preference learning system

The system learns from outfit choices using a weighted scoring model (not ML — intentionally simple for sparse data):

**Three data structures in `preferences.json`:**

1. `decisions` — raw log of every choice (append-only audit trail)
2. `item_scores` — per-item affinity to scenario features (e.g., "this jacket scores 1.3 for golf + winter"). Chosen items get +0.1 per matching feature; rejected items get -0.05.
3. `pair_scores` — how often two items are chosen together (e.g., "jacket + polo = 1.35"). Chosen pairs get +0.15.

**Safeguards:** All scores default to 1.0 (neutral, handles cold start). Scores clamped between 0.3–3.0 (prevents runaway or permanent burial). Users can reset preferences via UI.

**Learning status:** Cold start (0–5 decisions), building (6–20), well-trained (21+).

### Pre-filtering pipeline (for large wardrobes)

For a 150-item wardrobe, sending everything to Qwen would exceed useful context. The pipeline narrows before the API call:

1. **Season filter** — temperature heuristic: <10°C → winter, 10–20°C → fall/spring, >20°C → summer. Falls back to current month.
2. **Formality filter** — keyword matching on user message (e.g., "golf" → athletic/casual, "meeting" → business/formal).
3. **Item cap** — if still >60 items, truncate with balanced category representation.
4. **Compact format** — only send file, cat, sub, color, material, thick, desc, preference_score. Drop season/formality (already used for filtering).

### HEIC support

iPhone photos are HEIC by default. The cataloging pipeline handles this:

- `pillow-heif` registered at import time
- HEIC files included in the scan
- Converted to JPEG in memory before base64 encoding for the API
- Original HEIC preserved on disk for quality

### Sync mechanism (`sync.py`)

For ongoing wardrobe maintenance without re-cataloging everything:

- Compares files in `wardrobe_photos/` against entries in `wardrobe.json`
- New photos → catalog via Anthropic API
- Missing photos → remove from JSON
- Existing photos → skip
- Supports `--dry-run` (preview) and `--yes` (skip confirmation)
- 1-second delay between API calls to avoid rate limiting
- `catalog.py` also has resume support: skips already-cataloged items on re-run

## 3. Current user journey (non-technical household member)

### One-time setup (done by developer)

1. Clone repo, `pip install -r requirements.txt`
2. Create `.env` with both API keys
3. Start server: `uvicorn backend.app:app --host 0.0.0.0 --port 8000 --reload`
4. Share the LAN URL with household members (e.g., `http://192.168.1.50:8000`)

### Daily use (no coding required)

1. **Open browser** → go to bookmarked LAN URL (works on iPhone Safari)
2. **Select profile** → tap your name on the profile grid (or create one first time)
3. **Upload clothes** → "My Wardrobe" tab → "Upload Photos" → select from camera roll → progress bar shows cataloging
4. **Get outfit advice** → "Get Outfit Advice" tab → type scenario (e.g., "Golf tomorrow, 5°C in Tokyo") → receive outfit cards with photos
5. **Train preferences** → tap "Choose This Outfit" on the best option (or "Skip" to not train)
6. **Maintain wardrobe** → delete items in the grid, upload new ones, catalog button handles the diff

### Pain points in current version

- Initial server startup requires terminal access
- Web UI is functional but not polished
- Only accessible on home Wi-Fi (no remote access yet)

## 4. Planned changes

### Phase 1: UI polish (now)

**Goal:** Make the web UI feel like a real app, not a developer prototype.

**Approach:** Rework `frontend/index.html` with better visual design — improved layout, typography, spacing, transitions, mobile touch targets. Still a single HTML file, no build tools. May use an alternative AI tool for the design work.

**Not included:** Auto-launch script (will be configured on the target machine after code migration).

### Phase 2: Remote access via Tailscale (next)

**Goal:** Household members can access the app from anywhere, not just home Wi-Fi.

**Approach:** Install Tailscale on the host Mac and on each family member's device. Tailscale creates a private encrypted mesh network — the app gets a stable private IP (100.x.x.x) reachable from anywhere without exposing any public ports.

**Why Tailscale over alternatives:**

- Free for up to 100 devices
- End-to-end encrypted, no data passes through Tailscale's servers
- No port forwarding, no firewall configuration, no dynamic DNS
- Works across Wi-Fi networks, cellular, etc.
- Each device gets a stable IP — no URL changes

**Setup steps:**

1. Install Tailscale on host Mac (`brew install tailscale` or download from tailscale.com)
2. Install Tailscale on each family member's device (iOS app available)
3. All devices join the same Tailscale network (one admin account)
4. Access the app via `http://100.x.x.x:8000` from anywhere

### Phase 3: Machine migration (later)

**Goal:** Move the server off the development Mac onto a dedicated always-on machine.

**Considerations:**

- The entire project is self-contained in one folder — copy and run
- `.env` needs to be recreated on the new machine (gitignored)
- `users/` folder contains all user data — migrate this
- Set up auto-launch (launchd plist on macOS, systemd service on Linux)
- Tailscale will need to be installed on the new machine

## 6. Key technical decisions and rationale

**Flat JSON over database:** For 150 items and a few users, a JSON file is simpler to inspect, debug, backup, and version control than SQLite or Postgres. The entire wardrobe fits in <100KB of JSON.

**Heuristic season/formality detection over API call:** A regex-based keyword match is instant, free, and reliable enough for filtering. The AI model handles the nuanced styling decisions — the heuristic just reduces what it has to consider.

**Preference scoring over ML:** With 1–3 decisions per day, it takes months to accumulate enough data for gradient-based learning. A simple additive scoring system works immediately after 5–10 decisions and is fully transparent (you can read the scores in the JSON).

**Single HTML file over React/Vue build:** No npm, no webpack, no build step. The file can be edited with any text editor, served by any HTTP server, and understood by any AI coding tool. The tradeoff is less code organization — acceptable for a single-page app of this scope.

**Two API providers over one:** Cataloging (vision) is a one-time cost where quality matters most — Claude Sonnet is the best choice. Recommendations are frequent and text-only — a cheaper provider with an existing subscription makes sense. The OpenAI-compatible SDK means the recommendation provider is trivially swappable.
